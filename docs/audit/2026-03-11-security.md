# Security Audit — Research Toolkit

**Datum:** 2026-03-11
**Scope:** `src/`, `cli.py`, `pyproject.toml`, `.gitignore`
**Kontext:** Lokales CLI-Tool ohne Web-Exposition. Kein Server, keine User-Auth.

---

## 1. SECRETS

- [x] **Keine hardcoded API Keys** — Alle Keys via `os.environ.get()` geladen (S2, Exa, OpenAlex, LLM)
- [x] **`.env` in `.gitignore`** — Zeile 17: `.env` explizit ignoriert
- [x] **Keine Keys in Logs** — Logger-Meldungen enthalten nur Key-Namen (`S2_API_KEY nicht gesetzt`), nie Werte
- [x] **HTTPS erzwungen** — Alle API Base-URLs sind `https://` (OpenRouter, Semantic Scholar, Exa, OpenAlex)
- [ ] **MEDIUM: OpenAlex API Key in Query-Parametern** — `openalex_client.py:125` sendet `api_key` als URL-Parameter (`params["api_key"]`). Das ist OpenAlex-API-Konvention, aber Keys in URLs landen in Server-Logs und Browser-History. Bei einem CLI-Tool geringes Risiko, aber nicht ideal.
  - **Datei:** `src/agents/openalex_client.py:124-125`
  - **Empfehlung:** Kein Handlungsbedarf — OpenAlex API erwartet den Key als Query-Parameter, kein Header-Modus verfuegbar.
- [ ] **LOW: `_load_env()` setzt beliebige Env-Vars** — `cli.py:31-44` liest `.env` und setzt alle `KEY=VALUE` Paare via `os.environ.setdefault()`. Kein Whitelisting, aber da `.env` lokal und gitignored ist: akzeptables Risiko.
  - **Datei:** `cli.py:31-44`
  - **Empfehlung:** Optional: Whitelist auf bekannte Keys (`S2_API_KEY`, `EXA_API_KEY`, etc.)

## 2. INPUT SANITIZATION

- [x] **CLI File-Inputs validiert** — `document.exists()` Check vor `read_text()` in `review`, `check` Commands. `papers_file.exists()` Check im `search` Command.
- [ ] **LOW: Kein Path-Traversal-Schutz** — `--input`, `--papers`, `document` akzeptieren beliebige Pfade. Ein User koennte `../../etc/passwd` angeben. Da CLI-Tool vom gleichen User ausgefuehrt: kein reales Risiko (der User hat ohnehin Zugriff). Waere nur relevant bei Web-Wrapper.
  - **Dateien:** `cli.py:94-96`, `cli.py:196-198`, `cli.py:248-249`, `cli.py:284-285`
- [x] **BibTeX Parser: Kein Injection-Risiko** — `bibtex_parser.py` nutzt `bibtexparser` Library, extrahiert nur Strings (`title`, `author`, `doi`, `year`, `abstract`, `url`). Werte werden in Pydantic-Modelle validiert. Kein `eval()`, kein `exec()`, kein SQL. Manipulierte `.bib`-Dateien koennen maximal falsche Metadaten enthalten.
  - **Datei:** `src/utils/bibtex_parser.py`
- [x] **`OUTPUT_DIR` via `Path(...).mkdir()`** — Sichere Verzeichnis-Erstellung, kein Shell-Injection.

## 3. DEPENDENCY RISK

- [x] **Core Dependencies sauber** — `pydantic>=2.0`, `httpx>=0.27`, `rich>=13.0`, `typer>=0.12`, `bibtexparser>=1.4`. Keine bekannten CVEs (Stand Maerz 2026).
- [ ] **LOW: Floating Versions** — Alle Dependencies nutzen `>=` statt Pinning (`==`). Fuer ein Research-Tool akzeptabel, fuer Produktion waere ein Lockfile (`pip freeze > requirements.txt` oder `uv.lock`) besser.
  - **Datei:** `pyproject.toml:10-16`
  - **Empfehlung:** Optional: `uv lock` oder `pip-compile` fuer reproduzierbare Builds
- [ ] **LOW: Optional torch/sentence-transformers** — `[nlp]` Extra installiert `sentence-transformers>=3.0` + `numpy>=1.26`. PyTorch als transitive Dependency ist gross (~2GB), aber von PyPI/offiziellen Channels. Supply-Chain-Risiko minimal.
  - **Datei:** `pyproject.toml:28-31`

## 4. OUTPUT SAFETY

- [x] **API-Responses via Pydantic validiert** — Alle Clients nutzen `model_validate(response.json())` (SS, Exa, OpenAlex). Unerwartete Felder werden ignoriert, fehlende Felder haben Defaults.
- [x] **JSON-Output mit `ensure_ascii=False`** — Korrekte Unicode-Behandlung bei `json.dumps()`.
- [ ] **LOW: Kein XSS-Schutz in Markdown-Output** — `format_as_markdown()` in `forschungsstand.py:503-557` schreibt Paper-Titel und Autoren-Namen ungefiltert in Markdown. Ein manipulierter API-Response koennte `<script>` Tags enthalten. Nur relevant wenn Output in Web-UI gerendert wird (aktuell nicht der Fall).
  - **Datei:** `src/agents/forschungsstand.py:548-554`
  - **Empfehlung:** Kein Handlungsbedarf solange Output nur als Markdown/CLI konsumiert wird. Bei Web-Integration: HTML-Escaping einfuegen.
- [x] **LLM-Responses robust geparst** — `review_loop.py` und `query_generator.py` fangen `json.JSONDecodeError` ab, loggen Preview (max 50-200 chars), kein Crash bei ungueltigem Response.

## 5. ERROR HANDLING

- [x] **HTTP-Fehler geloggt ohne Credential-Leak** — `e.response.text[:200]` zeigt max 200 Chars Error-Body. Keine Keys in Error-Messages.
- [x] **Timeouts konfiguriert** — Alle Clients nutzen `timeout=30` (Sekunden). LLM-Client: `timeout_s=30.0`.
- [x] **Rate-Limit-Retry** — 429-Handling mit 1 Retry + 2s Delay bei allen API Clients.

## Zusammenfassung

| Priority | Count | Details |
|----------|-------|---------|
| CRITICAL | 0 | — |
| HIGH | 0 | — |
| MEDIUM | 1 | OpenAlex API Key in Query-Params (API-Konvention) |
| LOW | 4 | Path Traversal (CLI-only), Floating Versions, torch Supply Chain, XSS in MD-Output |

**Fazit:** Fuer ein lokales CLI-Tool ist das Security-Profil sauber. Keine hardcoded Secrets, alle API-Calls via HTTPS, Pydantic-Validierung auf API-Responses, robustes Error-Handling. Die LOW-Findings sind informativ und werden erst bei einer Web-Integration relevant.
